# v0.34 form-HITL 撞牆報告 — Codex briefing

> **2026-06-19 更新：✅ RESOLVED by Codex（v0.34.1）**
>
> **Root cause**：n8n Wait node 不同 `resume` mode 使用**不同的 URL 變數**：
> - `resume: webhook` → `$execution.resumeUrl` → 路徑 `/webhook-waiting/{id}`
> - `resume: form` → **`$execution.resumeFormUrl`** → 路徑 `/form-waiting/{id}`
>
> v0.34.0 開發時把 v2 Slack workflow 的 `$execution.resumeUrl` 直接拷貝到 v3 Form workflow 的 Email 連結 — POST 路徑變 `/webhook-waiting/`，但 form session 註冊在 `/form-waiting/`，互不接受對方的 payload → form submit 永遠失敗。
>
> **我的 4 輪繞路**（customData → staticData → hidden field → drop Respond + lastNode mode）**全部白工**，都在解錯誤的問題。Codex 一查 docker container access log 看到 POST 路徑直接認出 root cause。
>
> **Codex 給的完整修法（已 ship 至 v0.34.1）**：
> 1. Email 連結改用 `$execution.resumeFormUrl`
> 2. 入口 Webhook 改 `responseMode: onReceived` + `responseCode: 202`（POST 立即非同步回，不卡 caller）
> 3. 移除 Respond to Webhook 節點
> 4. Wait form 設 `responseMode: lastNode`
> 5. 修正無效的 `responseTimeout: "4h"` 為 `limitWaitTime: true / limitType: afterTimeInterval / resumeAmount: 4 / resumeUnit: hours`
> 6. 加 `hiddenField` 型 `correlationId`，用 `fieldValue: "={{ $('Stamp correlation').first().json.correlationId }}" `自動回填
> 7. **state 直接從 `$('Stamp correlation').first().json` 取得** — Wait form mode 會把 upstream runData 保存到 DB 並在 resume 時 reload，**根本不需要 customData/staticData**
>
> **環境**：n8n 2.10.3（不是 1.x — 我先前憑空假設錯）
>
> **Runtime 驗證（v0.34.1）**：
>
> | exec | round | status | svc /v1/void 呼叫 | Audit 分支 |
> | --- | --- | --- | --- | --- |
> | 526 | approve | success | ✅ 呼叫（svc 回 400 NOT_FOUND for fake invoice — 符合預期） | Audit (approved) |
> | 527 | reject | success | ❌ 正確跳過 | Audit (rejected) |
>
> 兩條分支都 end-to-end 跑通。Form HITL 路徑機制無誤。
>
> **教訓寫入**：[[feedback_n8n_resume_url_variables]] — 改 Wait `resume` mode 第一件事查當前 n8n 版本 docs 列出對應 URL 變數，不要從別的 mode 拷貝；遇到 form/webhook resume 失敗先看 docker access log 比看 executions API runData 快得多。
>

## 任務脈絡

要為 `examples/einvoice-n8n` 案例做出一個**台灣可用**的人在迴圈（HITL）作廢核可 workflow，避開 Slack（台灣使用率低）。Pack 內已存在三版：
- **v1（DIY）** — Slack 普通訊息塞 `$execution.resumeUrl` + Wait `resume:webhook` — 跑通過。
- **v2（Slack native）** — Slack `operation:sendAndWait` — 結構正確、Layer 1 roundtrip 通過，但需真 Slack workspace 才能 runtime 測。
- **v3（Form native，本檔重點）** — Email Send 通知 → Wait `resume:form` 提供 HTML 表單 → 用戶填表 → resume → svc /v1/void / audit。

v3 在 4 次迭代後仍**無法 runtime 跑通**。請 Codex 看「n8n Wait `resume:form` 跟 Webhook `responseMode` 怎麼搭配才正確」這條官方做法。

## v3 結構（運行版本）

```
Webhook: void request (POST /webhook/einvoice/void)
  └─> Stamp correlation (Code)
        └─> Email approver (HTTP → sandbox /email/send，含 $execution.resumeUrl)
              └─> Wait for form submission (resume:form, 4 fields: correlationId / decision / approver_email / note)
                    └─> Rehydrate after form (Code，讀 staticData[correlationId] 還原 state)
                          └─> Approved? (IF, 比較 $json.approved === true)
                                ├─[true]─> Call svc /v1/void (HTTP) → Rehydrate after void (Code) → Audit (approved, Sheet, disabled in sim)
                                └─[false]─> Audit (rejected, Sheet, disabled in sim)
```

## 已試過的設定組合與症狀

### 試法 A — Webhook responseMode = `responseNode` + Respond to Webhook 節點

- 觸發 webhook → POST 端 hang（webhook 等到 Respond 節點才回 caller）
- Email 送出 ✓，Wait 卡住等 form 提交 ✓
- 取 form URL `http://localhost:5678/webhook-waiting/{execId}` ← 注意：n8n 直接給的是 `n8n:5678` 內部 hostname，要手改 `localhost`
- **第 1 次 form submit** → 提交頁顯示：`Cannot assign to read only property 'name' of object 'Error: Node 'Stamp correlation' hasn't been executed'`
  - 根因：Rehydrate 用 `$node['Stamp correlation'].json` — Wait `resume:form` 後 n8n 把 runData 丟掉
- 改用 `$execution.customData.set/get('voidState', …)` 持久化 state
- **第 2 次 form submit** → 提交頁顯示：`voidState not in customData`
  - 根因：`customData` **也跨不過 Wait form boundary**（resume 後是新 execution context，customData 也清空了）
- 改用 `$getWorkflowStaticData('global')[correlationId] = state`，並在 form 加 hidden field `correlationId` 讓用戶手動 copy-paste（n8n Wait form 沒有 hidden / defaultValue 支援？）
- **第 3 次 form submit**（用戶手動貼 correlationId）→ 提交頁顯示：`Unused Respond to Webhook node found in the workflow`
  - exec runData：所有節點 OK ✅（Wait ok / Rehydrate ok / IF ok / Audit ok / Respond ok）但 `data.resultData.error.message = "Unused Respond to Webhook node found in the workflow"`
  - 解讀：workflow 邏輯實際跑完，但 n8n form 提交確認頁面跟 Respond to Webhook 衝突

### 試法 B — Webhook responseMode = `lastNode` + 刪掉 Respond to Webhook 節點

- 觸發 webhook → POST=200 立即回 ✓
- Email 送出 ✓，Wait 卡住等 form 提交 ✓
- **第 4 次 form submit**（用戶手動貼 correlationId、選 approve）→ 提交頁顯示：`Problem submitting response — Please try again or contact support if the problem persists`
- exec 525 從 executions API 查不到（list 只看到 522/523/524）— 直接 GET id=525 拿到 `status:"waiting"`、`finished:false`
- runData 卡在 Wait（"Wait for form submission" 還是 input 那筆 email POST 回應，沒被 form submit 觸發推進）

## 假設

1. n8n Wait `resume:form` 提供的 `/webhook-waiting/{execId}` URL **本身可能應該是 `/form-waiting/{execId}`**（form 專屬路徑）— n8n 1.x 不同小版本之間此路徑不一致？`$execution.resumeUrl` 給出的 URL 可能對 webhook 模式才正確，form 模式應該換成另一個變數？
2. n8n Wait `resume:form` 的 `formFields` 是否要在 `resume` 同層 sibling 而不是 `formFields.values` array？
3. n8n Wait `resume:form` 跟 Webhook `responseMode` 之間有明文限制 — 例如「Wait form 強制要求 webhook 用 responseNode + 且 Respond to Webhook 必須出現在所有 branch 結尾」？
4. n8n 1.x 是否有獨立的 **Form Trigger** + **Form Page** 節點對（不是 Wait form mode），才能正確處理 HITL 表單？官方 docs / blog 是否推薦此模式而非 Wait + form mode？
5. form submit 失敗是否是因為 form schema 包含 `correlationId` text field（無 defaultValue）— n8n 後端校驗看到 required 但 form 渲染時 user 沒填 → 拒絕 submit 並回 "Problem submitting"？
