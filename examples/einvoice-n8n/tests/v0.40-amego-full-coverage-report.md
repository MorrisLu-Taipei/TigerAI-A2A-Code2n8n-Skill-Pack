## 3. 修正過程中的真實發現（4 個）

跑這次 runtime 時連續發現我寫測試 input 用錯 SDK 欄位 / 用錯 enum / 缺必填，每次修正後重跑：

| 我先寫的（錯） | SDK 真實要求 | 影響 |
| --- | --- | --- |
| `buyer.taxId` | `buyer.ubn`（SDK schema 完全不接受 taxId） | A6 第一輪沉默 fallback 成 B2C — 表面 PASS 實質沒驗 B2B |
| `buyer.carrier` | top-level `carrier: { type, code }` | A9 第一輪沉默 fallback 成無載具 — 表面 PASS 實質沒驗 carrier |
| `buyer.loveCode` | top-level `donation: { npoban }` | A10 第一輪沉默 fallback — 表面 PASS 實質沒驗 donation |
| `taxType: "ZERO"` | `taxType: "ZERO_RATED"`（enum 明確）| A7 直接 reject 揭露 enum 錯誤 |
| `allowanceId` 太長 / 缺 | 必填且 `≤16 chars`（Amego 限制） | A3 reject |
| `providerOptions.customsClearanceMark` (camelCase) | `providerOptions.CustomsClearanceMark` (PascalCase MIG 原名 + 數字 1\|2) | A7 reject |

**最重要的元教訓**：跨多輪測試裡，**前一輪 8 個 PASS 中有 3 個是「偽 PASS」**（沉默 fallback 成 B2C）。**若沒有 SDK type definition 對照閱讀，僅憑「issue 200 + invoiceNumber」判定 PASS 會嚴重高估覆蓋**。這也是 §1.6 lexical schema-before-claim 規則想擋的事 — 沒 evidence 不可說「驗過」。

## 4. 真實 SDK 發現 — A12 SCHEDULED_ISSUE 不一致

### 4.1 發現

SDK 對 Amego 的 `capabilities[]` 列表（GET `/v1/capabilities/amego`）回：

```json
{
  "provider": "amego",
  "capabilities": [
    "ISSUE","VOID","ALLOWANCE","VOID_ALLOWANCE","QUERY",
    "B2B","MIXED_TAX","QUERY_BY_ORDER_ID","CARRIER_VALIDATION","FOREIGN_CURRENCY"
  ]
}
```

**SCHEDULED_ISSUE 不在列**。SDK README capability 對照表也明示 Amego 該欄位 `—`。

但實際送 `issue` 帶 `scheduledAt`：

```bash
curl -X POST /v1/issue -d '{
  "provider":"amego",
  "input":{ ..., "scheduledAt":"2026-06-27T00:00:00+08:00" }
}'
```

→ **Amego sandbox 接受並開立**（拿到真實 invoiceNumber `AA26515020`），未回 UNSUPPORTED。

### 4.2 推論

SDK 對 `capabilities[]` 是宣告性質、**沒有在 `issue()` runtime 強制執行**「不支援的欄位拒絕」。SDK 文件雖暗示「不支援的供應商會 throw UNSUPPORTED」但實際 path：
- `assertSupports(provider, SCHEDULED_ISSUE)` → 會 throw（**主動呼叫才會**）
- `provider.issue(input)` → 不會主動檢查 input 內 scheduledAt 是否合法 capability，直接 pass-through 給 Amego adapter，Amego 收到沒看懂的欄位 → 忽略，依平常 path 開立

### 4.3 影響評估

**對 Pack 案例的影響**：
- workflow `einvoice-scheduled-issue.workflow.json` 對 Amego 不會在 SDK 層被擋下 → 上游 caller 以為「Amego 預約成功」實際 Amego 立即開立。**容易誤導**。
- 補救路徑：(a) workflow 內呼叫 `einvoice-capability-aware-gate` 子 workflow 前置擋（v0.35.0 已 ship）; (b) 上 issue 個 PR 到 paid-tw/einvoice 建議在 issue() 內強制檢查 capability-marker fields。

**對 v0.40.0 release 的決定**：本報告附 **SEC-021 — SDK capabilities 宣告與 runtime 行為不一致** 條目到 SECURITY-REVIEW。建議使用 Pack 內建的 `einvoice-capability-aware-gate` workflow 作前置防線。

## 5. A10 DONATION PARTIAL 處置

Amego 接受 `donation: { npoban: "5168" }` 並開立 invoiceNumber `AA26515018`，但 raw response 不 echo `donation_mark` / `donation_npoban` 等欄位 → 無法從 response 證實「捐贈設定真的生效」。

**verification-method limitation**：要 100% 確認需：
- (a) Query 該發票 → 看 query response 是否有 donation marker
- (b) 登入 Amego 後台或財政部 platform 查
- (c) 等該 npoban 對應慈善機構通知

對 v0.40.0 評估：**邏輯路徑驗了**（SDK 接受、Amego 接受、開立成功），**結果生效驗未做**（Amego sandbox 不回該欄位）→ 標 PARTIAL 而非 FAIL。**生產環境**這條 path 需用上述 a/b/c 之一補驗。
